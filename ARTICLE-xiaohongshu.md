# 我给代码配了个 AI 文档管家 🤖 两种方案怎么选

> 程序员的老大难:代码天天改,文档永远是「上个版本」📦💀

最近折腾了个小工具:**研发提代码 PR,AI 自动判断要不要更新文档、出计划、人工点个赞同意后,它就把文档写好提 PR,你 review 留言它还会自己改** —— 中文为准,英文自动同步。

落地的时候有个有意思的岔路:**同一套能力,可以用两种完全不同的方式装进仓库。** 分享一下这两种方案和怎么选 👇

---

## 🅰️ 方案一:GitHub Actions(瘦触发器)

目标仓库里放一个几十行的小文件,核心逻辑全 `uses:` 一个**版本化的工具仓库**,跑在 GitHub 自己的服务器上。

✅ **零运维**,不用自己管服务器,白嫖 GitHub 算力
✅ 配置超简单,改逻辑只需发个新版本,各仓库 bump 一下 tag 就升级
⚠️ 每个仓库都得放那个文件
⚠️ 受 GitHub Token 限制 —— 机器人开的 PR 不会触发别的流程,想做「审核不过不让合」的硬卡就有点别扭

## 🅱️ 方案二:GitHub App + 自建后端(零文件)

目标仓库**一个文件都不放**,装个 GitHub App 就行。事件用 webhook 打到你自己的服务器 → 克隆仓库 → 跑同一套逻辑。

✅ **真·零文件**,仓库干干净净
✅ 一处部署,管所有仓库
✅ 不受 Token 那些限制,能做硬性审核门
⚠️ 得自己跑一个服务器(有运维成本)
⚠️ 要注册 App、管私钥和 webhook 密钥、还要个公网地址

> 最妙的是:**两套方案业务逻辑 100% 共享**,差别只在「触发 + 鉴权 + 拉代码」那一层薄薄的壳 🐚

---

## 🤔 到底怎么选

- 自己几个仓库 / 想快速验证 → 选 **A(Actions)**,十分钟跑通
- 给团队、组织铺开,追求「装上就用」的产品体验和更强控制 → 选 **B(App)**
- 我的路径:**先用 A 把核心逻辑验扎实,再外置成 B** ✅

---

## ✨ 顺便几个我觉得有意思的技术点

🔹 AI 改文档用「查找替换」做精确编辑,不整篇重写 → diff 超干净,reviewer 一眼看完
🔹 双语:中文为准、AI 自动同步英文,还专门让另一个 AI 当**译文质检员**,挑错译和「翻译腔」
🔹 判断「该不该改文档」看的是 **API 契约变没变**,不是文件动没动 → 纯内部重构它会主动说「不用改」
🔹 提速:翻译/质检用快模型、核心生成用强模型,一轮从 **8 分钟压到 2 分钟** ⚡
